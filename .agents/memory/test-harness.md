# Unit test harness

`pnpm test` runs `node --test test/` over plain `.mjs`/`.test.mjs` files. Because the
app's `lib/**` is TypeScript with `@/` path aliases, tests do NOT import `.ts`
directly. Instead the `pretest` script transpiles a hand-picked list of pure modules
with `tsc` into `.test-build/`, and tests import the emitted `.js`.

**Rule:** `tsc --rootDir X --outDir Y` mirrors the dir structure under `--rootDir`. One
`tsc` call = one rootDir. To test a module in a NEW lib subdir, append a SECOND `tsc`
invocation to `pretest` (chained with `&&`) with its own `--rootDir`/`--outDir`, then
import from the matching `.test-build/...` path.

- `pretest` only works for **pure** modules (no env/DB imports, no `@/` aliases that tsc
  can't resolve in isolation). `lib/ai/models.ts` and `lib/planner/*` qualify; route
  handlers / DB modules do not.
- `.test-build/` is gitignored — it's a build artifact, regenerated by `pretest`.
- The runner is registered as the `test` validation command (validation skill), so it
  runs as a CI-style gate via `startValidationRun(["test"])`.

**Why:** keeps tests dependency-light and fast without pulling Next.js/DB into the test
process; the price is that any new tested module must be added to a `tsc` line in
`pretest` or its import path won't exist.
